Design decisions — the questions evaluators ask

Why Chroma DB ? and not FAISS / Pinecone / Elasticsearch?
Embedded & zero-ops. Chroma runs in-process and persists to local disk — no database server to operate on the VPS, which keeps the project self-contained and $0. It integrates first-class with LangChain, supports metadata filtering and MMR natively, and lets us keep one collection per embedding model so the required open-source-vs-commercial comparison is a clean head-to-head.
  • FAISS — fast but lower-level: no built-in persistence or metadata ergonomics.
  • pgvector — needs a running Postgres.
  • Pinecone / Weaviate — hosted: network latency, API keys, cost → breaks zero-spend.
  • Elasticsearch / OpenSearch — great for keyword+vector at scale, but heavyweight to run; we get the keyword signal in-process via BM25 instead. (Open the Stack tab → Vector DB tile for the full comparison.)
Why hybrid retrieval ? instead of pure vector search?
Embeddings capture meaning but smear exact tokens — acronyms and rare terms like “NSP”, “BLEU”, “GPT-3”. BM25 ? nails those. Fusing both with RRF ? beat pure dense for every embedding in our benchmark, so bge + hybrid is the live default.
What are all the strategies — and what makes “Adaptive Hybrid” ours?
Seven, selectable live: dense, hybrid, hybrid_rerank ?, mmr ?, multi_query ?, hyde ?, and our own adaptive_hybrid: it reads the shape of your query (acronym / short / quoted → lean keyword; long / conceptual → lean semantic), sets the BM25↔dense weights accordingly, then applies MMR for diversity — with no extra LLM call. The Auto mode lets the query gate pick the strategy per question.
Why a pre-RAG query gate that asks clarifying questions?
Garbage-in, garbage-out: retrieving for “tell me about models” returns noise. The gate quality-checks the prompt before searching. If it's too vague it asks a clarifying question instead of guessing; if it's clear it routes to the best strategy (the keyword-vs-vector decision the mentors asked about). One cheap haiku call, and it fails open — any gate error just proceeds as a normal clear question.
How do you guarantee answers are grounded — no hallucination?
Three locks: (1) the generation prompt forbids answering outside the retrieved context (“say you don't know”); (2) the CRAG grader drops chunks that don't really help before generation; (3) every answer ships the exact source + page it used, and you can expand the trace to read the precise context sent to the model. If the papers can't cover it, it switches to web search and flags it 🌐 — it never silently invents.
Why run Claude locally via the CLI?
Generation, grading, the gate and the rewriter all go through the host's local claude CLI on the Claude Max plan — $0 per-query API cost, no API key in the app, and the papers never leave the box. The gate runs on the cheaper haiku model to stay fast.
Why did the cross-encoder reranker hurt here?
A cross-encoder ? is usually more accurate, but on this tiny 6-paper corpus it over-trims context to a few chunks and drops passages the answer needed — so it scored lower than plain hybrid in our benchmark. We keep it selectable (with real scores shown) for transparency, but it isn't the default. Honest results > a fancier-sounding pipeline.
How is multi-user memory isolated?
Each browser session gets its own id (cookie) and an isolated history row-set in SQLite. Follow-ups like “and what about BERT?” are condensed into a standalone question against that session's history before retrieval — context carries across turns without leaking between users.
